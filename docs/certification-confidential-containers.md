# Confidential Computing and Kata Containers Security: Comprehensive Research Report

## Executive Summary

Confidential computing has matured into a production-ready technology with **formal standardization through IETF RFC 9334 (RATS Architecture, January 2023) and RFC 9397 (TEEP Architecture, July 2023)**, comprehensive testing methodologies for all major TEE platforms, and explicit regulatory recognition as "state of the art" data protection. This report validates testing protocols across AMD SEV-SNP, Intel TDX, and ARM CCA platforms, maps confidential computing capabilities to major regulatory frameworks, and catalogs the rapidly evolving tooling ecosystem supporting attestation, key management, and security validation.

The most significant finding is that confidential computing directly addresses the critical gap in traditional security—**protecting data during active processing**—and has achieved regulatory recognition across GDPR (Articles 25, 32), HIPAA Security Rule, PCI DSS v4.0, SOC 2, FedRAMP, and emerging frameworks like NIS2 and DORA. As of 2025, **Kata Containers 3.19.1** provides production-grade confidential container support with active TEE integration, while a mature ecosystem of open-source tools enables comprehensive security testing and operations.

## Standards and certification frameworks establish foundation for secure implementation

The confidential computing standards landscape has solidified around **IETF specifications as the authoritative technical foundation**, with the Confidential Computing Consortium serving as an industry coordination body rather than a direct standards publisher. IETF RFC 9334 Remote Attestation Procedures (RATS) Architecture, published January 2023, defines the fundamental roles and conceptual messages for attestation: Attester, Verifier, Relying Party, Endorser, and Reference Value Provider. This architecture supports two key topological patterns—the passport model where attesters obtain attestation results to present to relying parties, and the background-check model where relying parties forward evidence to verifiers for validation.

NIST provides the most comprehensive U.S. federal guidance through the **IR 8320 series on Hardware-Enabled Security**. NIST IR 8320 (May 2022) explicitly covers confidential computing concepts, trusted execution environments, hardware security modules, and remote attestation services. The draft IR 8320D focuses specifically on hardware-based confidential computing for machine identity management and protection using TEEs. These publications represent formal acknowledgment that confidential computing addresses critical security requirements for federal systems. The NIST SP 800-53 Revision 5 control catalog includes **SI-16 Memory Protection** as a new control directly applicable to TEE memory encryption capabilities.

International standardization efforts center on **ISO/IEC AWI 25093-1 Cybersecurity - Confidential Computing**, currently in the Awaiting International Standardization stage. While this standard remains under development, existing ISO/IEC information security standards provide applicable frameworks. ISO/IEC 27001:2022 establishes information security management system requirements, ISO/IEC 27017:2015 provides cloud-specific security controls, and ISO/IEC 15408 (Common Criteria) offers security evaluation criteria applicable to TEE implementations.

The Confidential Computing Consortium has driven formal verification efforts, publishing research in IEEE Access 2024 on formal specification and verification of TEE attestation mechanisms in ARM CCA and Intel TDX using ProVerif symbolic security analysis. This work identified and reported security issues in Intel TDX TCB that were subsequently fixed, demonstrating the value of formal methods. The CCC Attestation Special Interest Group launched the Interoperable RA-TLS initiative in March 2023 to harmonize Remote Attestation TLS implementations across Gramine, RATS-TLS, Open Enclave, and SGX SDK projects, aligning with the DICE Attestation Architecture standard.

Certification frameworks for confidential containers have emerged with the **CNCF Confidential Containers project** achieving Sandbox status in March 2022. This project standardizes confidential computing at the pod/container level with hardware platform agnostic support for Intel TDX, AMD SEV, ARM CCA, and IBM SE. Professional certifications include the Certified Container Security Expert from Practical DevSecOps with 100% hands-on examination, the Linux Foundation's Certified Kubernetes Security Specialist requiring CKA prerequisite, and Sylabs' Singularity Containers Certification for HPC and scientific computing environments.

Microsoft Azure has positioned itself as a leader in attestation standards development, serving as active contributor to IETF RATS and TEEP working groups, document editor for RATS architecture, and co-chair of the CCC Attestation SIG. The Azure Attestation service provides a managed attestation platform demonstrating production implementation of these standards. This involvement ensures cloud provider implementations align with emerging interoperability standards.

## Testing methodologies provide comprehensive security validation across TEE platforms

AMD SEV-SNP testing centers on validating the Reverse Map Table for page ownership tracking, the eight-state page validation using the PVALIDATE instruction, and AES-GCM authentication for memory integrity with replay attack prevention. The **snpguest tool** (https://github.com/virtee/snpguest) serves as the primary command-line utility for managing AMD SEV-SNP attestation. Core testing procedures include SNP_INIT and SNP_INIT_EX for platform initialization, SNP_LAUNCH_START/UPDATE/FINISH for guest launch measurement, and SNP_GUEST_REQUEST for encrypted message handling with VMPCKs. Certificate chain validation follows ARK → ASK → VCEK/VLEK hierarchy, retrievable from AMD's Key Distribution Service at https://kdsintf.amd.com. The AMD SEV-SNP white paper and Firmware ABI specification (TechDoc #56860) provide authoritative testing guidance.

Testing limitations are explicitly documented: PRIME+PROBE cache attacks remain unmitigated, page-level access patterns are observable by the hypervisor, and physical DDR bus attacks fall outside the threat model. SMT (simultaneous multithreading) requires explicit policy-based opt-in through guest configuration, with guests able to require non-SMT platforms through the POLICY.SMT flag. Organizations must understand these boundaries when validating SEV-SNP security claims.

Intel TDX testing employs formal verification using the CBMC model checker for property checking of decomposed firmware and hardware subsystems. Memory encryption testing validates both CI mode with SHA-3-256 MAC per 64-byte cache line and LI mode with TD Owner bit checking. Intel's Memory Encryption Engine with Total Memory Encryption Key (MKTME) provides the cryptographic foundation. Critical testing procedures include TD Quote generation via the Quoting Enclave, PCK certificate retrieval from Intel Provisioning Certification Service, MRTD and RTMR measurement verification, and AEX-Notify validation for interrupt awareness to counter single-stepping attacks.

The **canonical/tdx project** (https://github.com/canonical/tdx) provides Ubuntu-specific TDX setup tools tested with v0.13.0 attestation components, including support for NVIDIA H100 GPU passthrough with Ollama AI workloads. Intel Trust Authority offers a production SaaS attestation service supporting TDX, SGX, TPM composite attestation, and preview support for AMD SEV-SNP and NVIDIA H100 GPUs. Intel's Security Research portal (https://www.intel.com/content/www/us/en/developer/articles/technical/software-security-guidance/technical-documentation/tdx-security-research-and-assurance.html) documents SDL penetration testing and coverage-guided fuzzing methodologies.

ARM CCA testing leverages **formal verification tools including HOL4 theorem prover** for Realm Management Monitor specification and CBMC for implementation verification. The ProVerif tool enables attestation protocol analysis, with formal specifications available at https://github.com/CCC-Attestation/formal-spec-TEE. ARM CCA's attestation architecture uses a platform token plus realm token structure in Entity Attestation Token format, cryptographically verified with Platform Attestation Key signatures. The **Veraison project** (https://github.com/veraison) provides the primary open-source attestation verification framework for ARM CCA, with evcli command-line tool, rust-ccatoken library, and cca-realm-measurements for computing RIM and REM values.

Testing must validate RMM lifecycle management, realm memory assignment via the Granule Protection Table, and transitions between Normal, Secure, and Realm worlds. Linaro operates a public verification service at https://veraison.test.linaro.org:8443 for pre-silicon CCA platforms using FVP emulation. ARM CCA represents newer technology with more limited field testing compared to Intel and AMD platforms, requiring careful evaluation of maturity and hardware availability.

Attestation testing frameworks follow the IETF RATS architecture with clear separation of Attestation Server as verifier, Attestation Client as evidence sender, Key Broker Service for secret management, and Key Broker Client for secret retrieval. Cross-platform frameworks include **Keylime** for runtime integrity monitoring, **Veraison** for multi-architecture verification, **Intel Trust Authority** for TDX/SGX attestation, and **Azure Attestation** for multi-TEE validation. Testing must verify both passport and background check models, validate nonce-based freshness, confirm certificate chain validation back to hardware roots of trust, verify TCB version checking, and compare measurements against reference values.

Memory encryption validation requires platform-specific testing approaches. AMD SEV-SNP testing validates private (C=1) versus shared (C=0) memory access using the C-bit in page tables, verifies VEK isolation per ASID, confirms AES-GCM authentication tag checking, and validates metadata integrity during page swapping. Intel TDX testing in CI mode verifies TD Owner bit validation, SHA-3-256 MAC verification with 28-bit tags, poison marking on integrity failures, and detection of wrong TD access attempts. ARM CCA testing validates GPT entry correctness, realm memory assignment enforcement, and RMM access control. Performance overhead typically ranges from **1-10% depending on workload characteristics**, measurable with tools like Intel Memory Latency Checker, STREAM benchmark, and Phoronix Test Suite.

Side-channel attack testing addresses interrupt-based, cache-based, and speculative execution attacks. AMD SEV-SNP provides optional BTB isolation with hardware tracking, SMT policy enforcement configurable by guests, and SPEC_CTRL MSR virtualization for IBRS. Testing validates BTB protection activation, hypervisor poisoning prevention, automatic flush on context switches, and SMT policy enforcement with SNP_ACTIVATE rejection on misconfigured systems. Intel TDX implements AEX-Notify for interrupt awareness, single-stepping countermeasures, and elapsed time detection. The **QuanShield research framework** demonstrates interrupt-free environment creation and tickless scheduling mode testing.

Side-channel assessment employs T-test based leakage assessment using Welch's t-test for resistance validation, with tools like ChipWhisperer, FOBOS (Flexible Open-source Board), and Riscure Inspector SCA. All platforms acknowledge limitations—microarchitectural attacks remain challenging, PRIME+PROBE falls partially out of scope, page-level monitoring cannot be fully prevented, and performance counter tracking remains exposed. Organizations must combine formal verification, penetration testing, side-channel analysis, and continuous monitoring for comprehensive validation.

Integrity measurement separates boot-time and runtime approaches. Boot-time measurement uses launch digests (AMD), MRTD with RTMR0-3 for firmware and kernel (Intel), and realm measurements via RMM (ARM). Runtime measurement integrates Integrity Measurement Architecture with policy configuration, measurement viewing, file verification, and TPM PCR extension. Composite attestation combining TEE and TPM provides independent measurement roots, collaborative reporting, and TPM-managed storage roots using tools like Microsoft TPM Simulator and CoCoTPM.

Key management testing validates complete key hierarchies. AMD SEV-SNP uses **CEK (Chip Endorsement Key) fused in hardware → VCEK (Versioned CEK with TCB version) → VLEK (Versioned Loaded EK from AMD KDS) → OEK (Offline Encryption Key for migration) → VEK (VM Encryption Key per guest) → VMPCK (four keys for guest-firmware communication) → VMRK (VM Root Key for derived keys)**. Intel TDX follows Intel Root CA → PCK Certificate → Attestation Key in TD Quoting Enclave → TD-specific derived keys. ARM CCA employs Platform Attestation Key, Realm Attestation Key, and KDF-derived sealing keys.

Key Broker Service testing scenarios validate attestation-gated key release, policy enforcement for TCB and measurements, key versioning and rotation, and audit logging. Production implementations include Azure Key Vault Managed HSM, Google Cloud KMS, AWS KMS with Nitro Enclaves, and HashiCorp Vault TEE integration. Cryptographic testing verifies FIPS 140-2/140-3 compliance, hardware RNG entropy, algorithm implementation correctness, performance under load, and side-channel resistance. Secure key release testing confirms valid certificate chains, TCB version minimums, measurement matching, policy compliance, and freshness validation while detecting invalid signatures, outdated TCBs, measurement mismatches, policy violations, and replay attacks.

## Kata Containers provides production-grade confidential computing with active development

Kata Containers has reached version **3.19.1 (January 2025)** with continuous security improvements addressing Rust dependency vulnerabilities. The architecture provides VM-based isolation with each container running in a lightweight virtual machine with dedicated kernel, leveraging Intel VT-x, AMD-V, ARM Hyp, IBM pSeries, and IBM Z SIE hardware virtualization. Core components include runtime-rs (Rust-based runtime providing containerd shimv2 implementation), kata-agent (management process inside the VM/POD), and support for QEMU, Cloud Hypervisor, Firecracker, and Dragonball hypervisors.

Recent security enhancements demonstrate active maintenance: version 3.19.0 updated libseccomp to v2.5.5 for improved seccomp filtering, enhanced memory security with rustix crate updates, addressed CVEs in libz-sys crate, and updated ring crate from 0.17.8 to 0.17.14. Version 3.18.0 added **TDX support in runtime-rs for Confidential Containers**, enhanced NVIDIA GPU support for confidential computing, improved policy management with Rust-based regorus crate replacing golang OPA, and updated kernels to 6.12.x series. Version 3.17.0 in May 2025 included high severity security bumps, RISC-V 64-bit architecture support in the agent, and VFIO-AP coldplug for s390x security enhancements.

Confidential computing implementation supports AMD SEV/SEV-ES/SEV-SNP with production support since early 2021, Intel TDX with production support in 3.18.0+, ARM CCA with planned support pending hardware availability, and IBM PEF for IBM Z/LinuxONE secure execution. Runtime classes include kata-qemu-snp for AMD SEV-SNP, kata-qemu-tdx and kata-clh-tdx for Intel TDX, kata-qemu-coco-dev for testing without specialized hardware, and kata-qemu-nvidia-gpu-snp for NVIDIA GPU with SNP support.

Configuration follows a simplified approach with confidentiality protection enabled dynamically based on hardware detection, image service offloading for encrypted container images, and built-in attestation support. The runtime automatically detects available confidential computing features via CPUID and enables the largest feature set without hard-coded requirements. Guest protection provides memory encryption and integrity when available, CPU-state confidentiality and integrity, and transparent enablement based on hardware capabilities.

Integration with Kubernetes uses the **Confidential Containers Operator** (https://github.com/confidential-containers/confidential-containers) installed via kubectl apply. Example pod configurations specify runtimeClassName: kata-qemu-snp with standard container definitions. The attestation architecture implements remote attestation protocol in kata-agent, guest enlightenment for measurement requests, quote building for tenant verification, and secure provisioning of encryption keys post-attestation.

Image service offloading enables encrypted container images with keys accessible only inside the guest, image layer decryption within the guest, CRI runtime seeing only encrypted layers, and prevention of host tampering with images. Security limitations are documented: AMD SEV-SNP lacks memory encryption (provides isolation), has no remote attestation in basic SEV mode, lacks secure DMA support, but supports VM migration. Intel TDX provides memory encryption and remote attestation, has secure DMA in specification, and supports VM migration in specification. ARM CCA lacks memory encryption (provides isolation), supports remote attestation, includes secure DMA in specification, and has undefined VM migration. IBM PEF lacks memory encryption (provides isolation), supports remote attestation, has no secure DMA support, lacks VM migration, but provides fully open-source implementation.

NVIDIA GPU confidential computing support integrates through the **GPU Operator version 24.9.0+** (https://docs.nvidia.com/datacenter/cloud-native/gpu-operator/24.9.0/gpu-operator-confidential-containers.html) with support for Intel TDX and AMD SEV-SNP. Installation deploys NVIDIA Kata Manager for managing kata artifacts, NVIDIA CC Manager for setting confidential computing mode on GPUs, NVIDIA Sandbox Device Plugin for discovering passthrough GPUs, NVIDIA VFIO Manager for loading vfio-pci driver, and Node Feature Discovery for detecting CPU security features.

Security testing infrastructure resides at https://github.com/kata-containers/tests with integration tests, CI tests, and metrics using the Bats (Bash Automated Testing System) framework. The **kata-runtime check** command validates hardware virtualization support, kernel modules, hypervisor availability, and network connectivity. The kata-collect-data.sh script gathers comprehensive system information for debugging. Agent testing tools include agent-ctl for low-level agent access and kata-ctl for advanced commands. GitHub Actions workflows provide build checks across x86_64, ARM, s390x, ppc64le, and RISC-V architectures, integration tests with containerd and CRI-O, confidential computing tests for TDX, SNP, and SEV, and security scanning with Zizmor, CodeQL, and Scorecard.

Known vulnerabilities have been systematically addressed through the Vulnerability Management Team process. **CVE-2020-2023 (CRITICAL)** allowed containers to access guest root filesystem device enabling code execution and kata-agent masquerading, requiring CAP_MKNOD capability to exploit, affecting versions 1.11, 1.10, 1.9, and fixed in 1.11.1+ and 1.10.5+ through device access restrictions. **CVE-2020-2024 (MEDIUM)** involved improper link resolution enabling unmounting arbitrary mount points on the host for denial of service, affecting versions prior to 1.11.0. **CVE-2020-2025 (HIGH)** in Cloud Hypervisor persisted guest filesystem changes to host images allowing malicious guests to overwrite image files and control subsequent VMs, affecting versions prior to 1.11.0.

Recent security fixes in January 2025 addressed unsafe-libyaml, zerocopy, ring, and libz-sys vulnerabilities. May 2025 patched high severity Rust vulnerabilities, golang.org/x/net issues, openssl updates from 0.10.57 to 0.10.72, and tokio security updates. The vulnerability management process requires private bug reporting via Launchpad (not public GitHub issues), VMT coordinator assignment, KCSA drafting, CVE assignment, fix development and review, downstream stakeholder notification, and public disclosure with patches.

Security considerations include known limitations: partial seccomp support limited to --security-opt=no-new-privileges and seccomp profiles, no AppArmor support, SELinux not yet implemented, network namespace sharing not supported with traditional containers, and resource constraints requiring careful VM-based architecture management. Attack surface analysis shows reduced host kernel exposure from containers, additional hypervisor/VMM in the trusted computing base, but mitigated through minimal guest kernel reducing attack surface.

Best practices for secure deployments include hardware virtualization verification, RuntimeClass configuration with appropriate resource overhead (160Mi memory, 250m CPU), workload annotations for untrusted code using io.kubernetes.cri.untrusted-workload: "true", security hardening with enable_guest_seccomp, minimal kernel configuration, capability dropping, guest seccomp building with SECCOMP=true flag, network security with vhost-net, and comprehensive monitoring through Datadog Agent v7.51+ or centralized logging with kata-agent journald forwarding.

Production deployments include Ant Group with large-scale deployment of thousands of containers combining Kata isolation with scheduling and LLC isolation, Baidu for function computing and cloud container instances, Huawei for CCI and CCE turbo, and IBM Cloud for secure shell and CI/CD pipelines. AWS EKS deployment uses bare metal instances with eksctl and kata-deploy for automated installation. The monthly release cadence for 3.x versions ensures continuous security updates with libseccomp notices, builder images, and detailed changelogs.

## GDPR compliance achieved through technical measures and data protection by design

Confidential computing has achieved explicit recognition as **"state of the art" technology for data protection compliance** by ENISA and TeleTrusT as of 2024. Article 32 GDPR mandates appropriate technical and organizational measures to ensure security appropriate to risk, specifically including pseudonymisation and encryption of personal data (Article 32(1)(a)) and ongoing confidentiality, integrity, availability, and resilience of processing systems (Article 32(1)(b)). TEEs satisfy these requirements by providing hardware-enforced encryption of data during active processing in memory, addressing the critical gap in traditional security that only protects data at rest and in transit.

The encryption requirement extends beyond storage and transmission. Memory encryption ensures data remains encrypted even during computation, with keys managed within secure enclaves inaccessible to operating systems, hypervisors, or cloud providers. Hardware-isolated execution environments prevent unauthorized access even by privileged users, meeting the ongoing confidentiality requirement. Code and data integrity verification through attestation mechanisms satisfies integrity requirements. Processing continues securely even in compromised host environments, demonstrating resilience. Protection against privileged malware, root access attacks, and insider threats addresses the comprehensive risk assessment required by Article 32(2) covering "accidental or unlawful destruction, loss, alteration, unauthorised disclosure of, or access to personal data."

Article 25 GDPR requires data protection by design and by default, with EDPB Guidelines 4/2019 (Version 2.0, October 2020) emphasizing continuous assessment of "state of the art." Confidential computing represents current technological advancement in data protection with hardware-based security providing the highest level of protection available. Default configuration ensures only authorized code can access data in TEEs, with attestation mechanisms verifying processing occurs only in trusted environments, satisfying data protection by default requirements.

International data transfers under Articles 44-50 GDPR present particular challenges post-Schrems II. EDPB Recommendations 01/2020 on supplementary measures specifically address cloud computing in Use Case 6. Confidential computing provides encryption during processing in third countries, key management under EU controller control, attestation proving processing in genuine TEEs, and hardware-enforced isolation preventing cloud provider access. This addresses the technical impossibility of government access without encryption keys, eliminates backdoors through hardware-based security, and provides verifiable security through remote attestation.

Recent regulatory recognition includes **EU DORA (Digital Operational Resilience Act)** mandating data-in-use protection for financial institutions effective January 17, 2025, NIST CSF 2.0 adding guidance on protecting data-in-use in February 2024, and the Federal Zero Trust Guide from US Federal CISO Council explicitly mentioning confidential computing. The EuroCloud Deutschland whitepaper "Confidential Computing: Sicher und souverän in der Cloud" (December 2024) confirms compliance with GDPR, DORA, and NIS2. Technology receives explicit mention in EDPB supplementary measures recommendations as technical safeguard for international transfers.

## HIPAA Security Rule requirements satisfied through comprehensive technical safeguards

The HIPAA Security Rule at 45 CFR §164.312 establishes technical safeguards that confidential computing directly addresses. Access control requirements under §164.312(a) mandate technical policies for access control, unique user identification, and addressable encryption and decryption specifications. Hardware-enforced access control prevents privileged user access to ePHI, TEE attestation verifies identity before granting data access, and continuous encryption including during processing exceeds NIST SP 800-111 and 800-52 standards by adding in-use protection.

Audit controls under §164.312(b) require recording and examining activity in systems containing ePHI. Immutable audit logs within TEEs cannot be tampered with by administrators, comprehensive activity recording captures all ePHI access, and attestation records provide cryptographic proof of execution. Integrity requirements under §164.312(c) mandate protection of ePHI from improper alteration or destruction. Code integrity verification through TEE attestation, hardware-based memory encryption preventing unauthorized modifications, and tamper-proof processing in isolated execution satisfy these controls.

Person or entity authentication under §164.312(d) requires verifying identity of persons or entities seeking ePHI access. Multi-factor authentication integrates with TEEs, cryptographic identity verification occurs through attestation, and hardware-rooted identity via TPM integration provides strong authentication. Transmission security under §164.312(e) with addressable encryption specifications receives comprehensive coverage through end-to-end protection combining TLS in transit, AES at rest, and TEE protection during use, ensuring complete lifecycle protection with no unencrypted states meeting all NIST standards.

Recent HIPAA guidance published in the Federal Register (January 6, 2025) on "HIPAA Security Rule To Strengthen Cybersecurity" increases emphasis on encryption as best practice, recognizes that encryption is "built into most software today," employs a risk-based approach acknowledging evolving threats, and aligns with proposed enhanced requirements that confidential computing satisfies. OCR enforcement actions increasingly focus on data-in-use protection in audits, making TEE adoption a proactive compliance measure.

The addressable specification nature of HIPAA encryption requirements means organizations must implement encryption or document reasonable alternative measures. Confidential computing provides the most comprehensive encryption approach available, potentially serving as safe harbor for breach notification requirements under 45 CFR §164.402(2) where encrypted data may not constitute a breach if decryption keys remain secure.

## PCI DSS v4.0 addresses data-in-use protection with confidential computing solutions

PCI DSS version 4.0 became mandatory March 31, 2024 with version 3.2.1 retired, introducing **explicit requirements for protecting data during processing including volatile memory**. Requirement 3 protects stored account data with sub-requirement 3.4.2 mandating PAN rendered unreadable anywhere stored, 3.5.1 requiring technical controls for PAN protection, and new requirement 3.5.1.1 demanding technical controls prevent PAN copying or relocation. Confidential computing satisfies these through ephemeral processing of sensitive authentication data in TEEs without persistent storage, hardware isolation preventing unauthorized PAN copying, secure key storage and hashing within TEEs, TEE memory encryption complementing disk encryption for defense in depth, and hardware-protected key generation and storage.

Requirement 4 protects cardholder data during transmission with 4.2.1 requiring strong cryptography and new sub-requirement 4.2.1.1 specifying industry-accepted protocols (TLS 1.2+, IPsec, SSH). Confidential computing provides end-to-end protection combining TLS with TEE encryption ensuring PAN never exists unencrypted outside secure enclaves, mutual TEE attestation before key exchange, and certificate private key protection in hardware.

Requirement 8 on identity and authentication includes future-dated requirement 8.3.6 (mandatory March 2025) for minimum 12-character passwords and 8.3.10.1 requiring MFA for administrative access. Confidential computing enhances these controls through hardware-backed MFA via FIDO2/WebAuthn in TEEs, secure biometric template storage, password verification without exposure, and protected session token management.

Requirement 10 on logging and monitoring specifies 10.3.4 requiring log entries include user ID, event type, date/time, and success/failure indicators. Confidential computing satisfies requirements through immutable logs generated within TEEs, cryptographic sealing preventing tampering, protected log storage and transmission, and administrator-proof logging that prevents privileged users from modifying audit trails.

PCI DSS v4.0 specific updates emphasize enhanced focus on data-in-use with Requirement 3.4.2 now emphasizing protection during processing, version 4.0.1 adding volatile memory (RAM) protection requirements, memory dump file protections explicitly required, and confidential computing directly addressing these enhancements. The new Customized Approach allows alternative controls meeting security objectives, with organizations demonstrating equivalent or better protection. **Confidential computing qualifies as a customized approach** requiring attestation reports, testing, and security audits as evidence.

PCI SSC explicitly recognized in December 2024 that "PCI DSS v4.0.1 includes several references to protecting data-in-use, including guidance on volatile memory like RAM as well as memory dump files," directly addressing the use case for confidential computing in payment processing environments. Organizations handling cardholder data in cloud environments or with complex data flows should evaluate TEE adoption to satisfy these enhanced requirements.

## SOC 2 and FedRAMP requirements map directly to confidential computing controls

SOC 2 Trust Service Criteria from AICPA (2017, Updated 2022) establish control objectives that confidential computing satisfies across multiple categories. The CC6 family on logical and physical access controls includes **CC6.1 requiring security measures against threats from outside system boundaries**. TEEs enforce hardware-isolated access controls preventing unauthorized access even from privileged users, cloud operators, and OS administrators through memory encryption, attestation mechanisms, and secure enclaves. CC6.6 protecting information assets from security threats receives satisfaction through runtime memory encryption ensuring data remains protected during processing, addressing insider threats and physical access attacks. CC6.7 restricting access to authorized internal and external parties leverages TEE attestation verifying code integrity before granting access to encrypted data.

The CC7 family on system operations includes CC7.2 requiring detection policies and tools to identify anomalies, satisfied through TEE platform continuous monitoring and attestation capabilities detecting tampering or unauthorized modifications. CC7.3 evaluating security events for potential failures receives support from secure audit logging within TEEs providing tamper-proof event records for security analysis. The C1 confidentiality criteria require identifying and maintaining confidential information (C1.1) and disposing of confidential information properly (C1.2). Data encrypted in-use within TEEs maintains confidentiality during active processing, while secure data destruction within TEE memory regions ensures no residual data leakage. PI1 processing integrity criteria demand complete, accurate, and timely output (PI1.4), satisfied through TEE code integrity verification ensuring processing occurs without tampering.

FedRAMP requirements based on **NIST SP 800-53 Revision 5** (September 2020, Updated December 2020) establish comprehensive security controls applicable to cloud service offerings. The System and Communications Protection (SC) family includes critical controls: SC-4 preventing information transfer via shared system resources through hardware isolation, SC-7 boundary protection with SC-7(4) for external telecommunications services using TEE cryptographic boundaries, **SC-8 transmission confidentiality and integrity with SC-8(1) cryptographic protection** required at all FedRAMP baselines and satisfied through end-to-end encryption within TEE, SC-12 cryptographic key establishment and management supported by TEE-based key management with HSM integration, SC-13 implementing FIPS-validated cryptography with TEE platforms supporting FIPS 140-2/140-3 validated modules, and **SC-28 protection of information at rest with SC-28(1) cryptographic protection** extended by TEEs to data-in-use.

The Access Control (AC) family includes AC-3 access enforcement with AC-3(9) controlled release where TEE attestation ensures data release only to verified authorized code, AC-4 information flow enforcement with AC-4(2) processing domains created through TEE hardware isolation, and AC-6 least privilege with AC-6(9) logging privileged functions captured in tamper-resistant audit logs. The Audit and Accountability (AU) family mandates **AU-9 protection of audit information with AU-9(3) cryptographic protection** for High baseline, satisfied through audit logs generated within TEE being cryptographically protected and sealed, plus AU-10 non-repudiation through TEE attestation providing cryptographic proof of code execution and data processing.

The System and Information Integrity (SI) family requires **SI-7 software, firmware, and information integrity with SI-7(1) integrity checks and SI-7(6) cryptographic protection** through TEE remote attestation verifying software integrity before execution, plus the new **SI-16 memory protection control** directly addressed by hardware memory encryption in TEEs. FedRAMP Moderate baseline (323 controls) benefits from TEE satisfaction of 40+ controls related to cryptography, access control, and data protection including SC-8, SC-13, SC-28, AC-3, AU-9, and SI-7. High baseline (410 controls) requires TEE for enhanced protection with additional controls SC-8(1), SC-28(1), AU-9(3), SI-7(1), and SI-7(6).

Australian Prudential Regulation Authority CPS 234 Information Security (July 2019) with CPG 234 guidance establishes requirements for APRA-regulated entities. Paragraph 20 on information asset classification by criticality and sensitivity enables TEE secure processing of highly sensitive data like biometric data and financial transactions. **Paragraph 21 mandates controls commensurate with vulnerabilities, threats, criticality, sensitivity, life-cycle stage, and potential consequences**, comprehensively addressed through TEE hardware isolation and encryption across all dimensions.

Paragraph 35 APRA notification requirements mandate 72-hour incident reporting with TEE reducing incident likelihood through hardware-protected environments. CPG 234 Section 9.1.6 guidance states "Confidential information stored on IT systems, servers and databases should be encrypted and protected through strong access controls" with TEE extending encryption to runtime environments.

## NIS2 Directive and DORA mandate enhanced cybersecurity measures

The NIS2 Directive (EU 2022/2555) entered into force January 16, 2023 with Member State implementation deadline of October 17, 2024. **Article 21 establishes cybersecurity risk management measures** requiring technical and organizational controls. Article 21(1)(a) on risk analysis and information system security policies receives support from hardware-based risk mitigation for data processing risks. Article 21(1)(d) on supply chain security leverages TEE attestation verifying integrity of software supply chain. **Article 21(1)(g) explicitly requires cryptography and encryption**, satisfied through TEE hardware encryption for data-in-use complementing at-rest and in-transit encryption to complete the CIA triad.

Article 21(1)(h) on human resources security, access control, and asset management benefits from hardware access controls preventing insider threats. Article 21(1)(i) requiring multi-factor authentication or continuous authentication receives implementation through TEE attestation as cryptographic authentication mechanism. Article 23 reporting obligations mandate early warning within 24 hours, notification within 72 hours, and final report within one month, with TEE reducing incident frequency through enhanced security controls and audit logs supporting comprehensive incident reporting.

NIS2 sector coverage spans 18 sectors including banking and financial market infrastructures using TEE for secure transaction processing, healthcare for protected health information processing, cloud computing service providers with TEE as core security capability, and digital infrastructure for secure network functions. ENISA technical guidelines (2024) emphasize hardware security modules and TEEs for critical operations, guidance on confidential computing for multi-cloud strategies, and integration with EU Cybersecurity Certification Schemes including Common Criteria certification for TEE platforms and Cloud Services Cybersecurity Certification Scheme (EUCS) including TEE evaluation criteria.

The Digital Operational Resilience Act (Regulation EU 2022/2554) becomes effective January 17, 2025, requiring Article 9 protection and prevention through TEE hardware security for financial transaction processing, Article 11 multi-cloud strategy requirements enabled through secure workload portability across clouds, and Articles 28-30 third-party ICT risk management supported by TEE attestation for vendor security validation. The overlap between DORA and NIS2 creates comprehensive cybersecurity requirements for EU financial institutions that confidential computing directly addresses.

## Tools ecosystem provides comprehensive testing and operational capabilities

Attestation verification tools form the foundation of confidential computing security. For AMD SEV-SNP, **snpguest** (https://github.com/virtee/snpguest) serves as the primary CLI utility for managing attestation, requesting reports with VMPL support, managing certificates (ARK, ASK/ASVK, VCEK/VLEK), and integrating with AMD Key Distribution Service. Installation requires Rust toolchain with build via cargo build -r. The **go-sev-guest** library (https://github.com/google/go-sev-guest) developed by Google wraps /dev/sev-guest device for quote generation, certificate chain validation, CRL checking, and policy-based attestation validation. AMD provides **sev-guest reference implementation** (https://github.com/AMDESE/sev-guest) demonstrating end-to-end remote attestation with SSH key exchange and encrypted disk unlock examples.

Intel TDX attestation centers on **Intel Trust Authority** (https://docs.trustauthority.intel.com/), a production SaaS attestation service supporting Intel TDX using go-tdx adapter, Intel SGX, TPM/vTPM with composite attestation, Azure confidential VMs via go-aztdx, GCP confidential VMs, preview support for NVIDIA H100 GPUs, and limited preview for AMD SEV-SNP. The service provides RESTful API endpoints, JWT attestation tokens, policy-based verification, and multi-platform support requiring Go 1.22+. Intel TDX DCAP Libraries (https://download.01.org/intel-sgx/latest/dcap-latest/) include TDX Quote Generation Library, Quote Verification Library, and TDQE (TD Quoting Enclave) depending on Intel SGX SDK and DCAP infrastructure. Canonical provides **canonical/tdx** (https://github.com/canonical/tdx) tested with v0.13.0 attestation components for Ubuntu with remote attestation integration and NVIDIA H100 GPU support.

ARM CCA attestation leverages the **Veraison Project** (https://github.com/veraison), an open-source attestation verification framework donated to the Confidential Computing Consortium. The **evcli** tool (https://github.com/veraison/evcli) provides command-line manipulation of attestation tokens with CCA token operations, attester and relying-party modes, and challenge-response protocol support requiring Go 1.23+ and jq utility. **rust-ccatoken** (https://github.com/veraison/rust-ccatoken) implements ARM CCA attestation tokens per RMM Specification §A.7 with token verification using CPAK, golden value extraction, and appraisal operations. **cca-realm-measurements** (https://github.com/veraison/cca-realm-measurements) computes Realm Initial and Extensible Measurements generating RIM and REM for attestation with CoRIM file generation and VMM support for kvmtool and cloud-hypervisor. Linaro operates an experimental verification service at https://veraison.test.linaro.org:8443 for pre-silicon CCA platforms.

Key Broker Service implementations center on **Trustee** (https://github.com/confidential-containers/trustee), formerly CoCo KBS, at version v0.13.0 (2024) providing attestation and secret delivery for Confidential Containers. Architecture components include Key Broker Service facilitating remote attestation and secret delivery as Relying Party in RATS model, Attestation Service verifying TEE evidence as Verifier, Reference Value Provider Service managing reference values for verification, and KBS Client Tool for configuration and testing. Supported TEEs include Intel SGX, Intel TDX, AMD SEV-SNP, and IBM Secure Execution with deployment via Docker Compose, Kubernetes through KBS Operator, or Helm charts. Integration works with Confidential Containers Attestation-Agent and Confidential Data Hub, implemented in Rust and Go as core component of CNCF Confidential Containers project.

Library OS and TEE frameworks enable running applications in secure enclaves. **Gramine** (https://github.com/gramineproject/gramine) at version v1.4 (2023) with v1.5 adding TDX support provides a library OS for running unmodified Linux applications in Intel SGX/TDX under LGPL-3.0 license. Key features include multi-process support, RA-TLS (Remote Attestation TLS), Secret Provisioning libraries, EDMM support from v1.4+, file system encryption, and manifest-based configuration. Installation via .deb and .rpm packages requires Intel SGX SDK/driver and Python 3.6+, with examples including Redis, Nginx, GCC, Python, and Node.js in production-ready status.

**Enarx** (https://github.com/enarx/enarx) from the Confidential Computing Consortium provides TEE-agnostic runtime for WebAssembly applications supporting Intel SGX and AMD SEV-SNP with future ARM CCA and Intel TDX support. Key features include WebAssembly-based runtime, language support for Rust, C/C++, C#, Go, Java, Python, and Haskell, CPU-architecture independence, hardware vendor neutrality, and single attestation framework. Installation via binary packages (rpm, deb) with CLI commands enarx init, build, run, sign, and key implemented in Rust with 1.4k+ GitHub stars demonstrating active development.

**Occlum** (https://github.com/occlum/occlum) at version v0.30.0+ provides memory-safe, multi-process LibOS for Intel SGX written in Rust under BSD License. Originally developed by Ant Financial, the open-source project achieves 1000x faster process creation and 3x faster IPC than traditional SGX LibOSes through single-address-space architecture. Features include multiple file system support (hashed FS, encrypted FS, host FS), EDMM support for flexible configuration, and JSON-based configuration with publication at ASPLOS'20.

**MarbleRun** (https://github.com/edgelesssys/marblerun) from Edgeless Systems provides service mesh / control plane for confidential computing with multi-process-per-enclave coordinator. Key features include remote attestation for SGX enclave clusters, key management and secret provisioning, mTLS connection setup, secure recovery mechanisms, manifest-based JSON deployment, and single attestation statement for entire deployment. Supported runtimes include Gramine, EGo, and Occlum with Kubernetes SGX-enabled support, standalone deployment, Client REST API (port 4433), and Mesh API (port 2001) implemented in Go with Linkerd service mesh compatibility providing production-ready status.

Benchmarking and performance testing employs multiple open-source tools. **k6** (https://github.com/grafana/k6) from Grafana Labs under AGPL-3.0 license with 27.2k+ GitHub stars provides modern performance testing for APIs, microservices, and web apps using JavaScript-based test scripting, HTTP/2, WebSocket, and gRPC support, Kubernetes fault injection, metrics export to DataDog, Prometheus, and NewRelic, browser performance testing via xk6-browser extension, and cloud-native CI/CD integration. **Apache JMeter** (https://jmeter.apache.org/) from Apache Software Foundation under Apache 2.0 license (first release 1998) supports load testing and performance measurement across HTTP/HTTPS, FTP, JDBC, LDAP, SOAP/REST, and TCP protocols with multi-threaded framework, GUI and CLI modes, distributed testing, extensibility via plugins, and real-time results and reporting.

Additional performance tools include **wrk** for modern HTTP benchmarking with multi-threading, multi-core CPU leveraging, Lua scripting support, and high-performance C implementation; **Apache Bench (ab)** for lightweight Apache HTTP server benchmarking measuring requests per second, response time analysis, and throughput evaluation; **Siege** (https://github.com/JoeDog/siege) with 6.1k+ GitHub stars for HTTP load testing simulating concurrent users; and **Tsung** (https://github.com/processone/tsung) with 2.6k+ stars providing distributed Erlang-based load testing for HTTP, XMPP, LDAP, MQTT, and WebSocket protocols with massive concurrent user simulation and XML-based scenario configuration.

Monitoring and audit tools include **Datadog Agent v7.51+** (2023) providing observability platform with Kata Containers support featuring automatic metrics, traces, and logs collection, anomaly detection through Watchdog, Docker/Kubernetes label import, container tagging and filtering, and performance monitoring requiring Datadog Agent installation for production monitoring of confidential workloads.

## Gap analysis reveals opportunities for enhanced testing and compliance frameworks

Missing testing protocols include comprehensive side-channel attack testing frameworks publicly available for all TEE platforms beyond research tools, standardized performance benchmarking suites comparing overhead across AMD SEV-SNP, Intel TDX, and ARM CCA under identical workloads, automated fuzzing frameworks specifically designed for TEE boundary interfaces, and formal verification toolchains accessible to organizations beyond academic researchers and chip vendors. Organizations currently rely on vendor security advisories and academic research rather than standardized testing methodologies, creating opportunities for industry consortia to develop shared testing frameworks.

Additional regulatory frameworks warranting coverage include **Healthcare sector-specific:** FDA Software as Medical Device guidance incorporating confidential computing for medical device security; **Federal systems:** DoD Impact Level 4-5 requirements and FedRAMP High+ provisional authorizations requiring enhanced protection; **Industry-specific:** SWIFT Customer Security Programme incorporating TEE recommendations for payment processing; **International:** China's Personal Information Protection Law, Brazil's LGPD, India's Digital Personal Data Protection Act mapping to confidential computing capabilities; **Emerging regulations:** EU AI Act technical requirements for AI system security and Canadian Consumer Privacy Protection Act data protection standards.

Essential tools not yet widely available include automated policy generation tools translating organizational security policies into TEE attestation policies and Key Broker Service access controls, continuous compliance monitoring platforms automatically verifying TEE configurations against regulatory requirements, integrated vulnerability scanning for confidential computing stacks detecting misconfigurations and known weaknesses, and forensic analysis tools enabling security incident investigation within TEE constraints while preserving confidentiality properties.

Current framework gaps include insufficient guidance on TEE selection criteria mapping specific workload requirements to appropriate TEE technologies, limited documentation on migrating existing applications to confidential computing environments with minimal code changes, inadequate multi-TEE deployment strategies enabling workload portability across AMD, Intel, and ARM platforms, and missing integration patterns for hybrid environments combining TEEs with traditional security controls. Industry collaboration through the Confidential Computing Consortium and CNCF Confidential Containers project addresses some gaps, but comprehensive best practices remain under development.

## Strategic recommendations for testing protocol and compliance implementation

Organizations should prioritize establishing comprehensive TEE testing programs beginning with attestation validation as the foundation of security. Implement automated certificate chain verification testing back to hardware roots of trust, develop freshness testing for nonce-based challenge-response protocols, create measurement validation against known-good reference values, and establish TCB version checking ensuring minimum security baselines. Deploy continuous attestation monitoring detecting configuration drift or evidence of tampering, with integration into security information and event management (SIEM) platforms for centralized visibility.

Memory encryption testing should follow platform-specific methodologies validating encryption boundaries between private and shared memory regions for AMD SEV-SNP using C-bit verification, confirming VEK isolation per ASID, testing AES-GCM authentication tag checking, and validating metadata integrity during page operations. For Intel TDX, verify TD Owner bit enforcement, test SHA-3-256 MAC validation per cache line in CI mode, confirm poison marking on integrity failures, and validate protection against cross-TD access. ARM CCA testing validates GPT entry correctness, realm memory assignment enforcement, and RMM access control with transition testing between security worlds.

Regulatory compliance implementation demands mapping existing controls to TEE capabilities through control matrix development identifying which NIST 800-53, ISO 27001, or other framework controls receive enhancement through confidential computing. Document TEE implementation in System Security Plans for FedRAMP, update Data Protection Impact Assessments for GDPR including TEE technical measures, revise Business Associate Agreements for HIPAA addressing TEE data processing, and prepare customized approach documentation for PCI DSS demonstrating equivalent or superior protection.

Organizations should pursue certification and audit preparation engaging assessors familiar with confidential computing early in planning, providing attestation reports and verification evidence as compliance artifacts, documenting continuous monitoring and testing procedures, and maintaining comprehensive configuration baselines. Third-party security testing through penetration testing of TEE implementations, formal verification where critical (using tools like ProVerif, CBMC, HOL4), side-channel analysis using T-test methodologies and specialized equipment, and regular vulnerability scanning of TEE software stacks provides assurance.

Tool selection and deployment should prioritize production-ready platforms including Gramine v1.4+ for Intel SGX/TDX applications, Confidential Containers for Kubernetes environments (CNCF sandbox project), Intel Trust Authority or Azure Attestation for attestation verification, and Trustee/KBS v0.13.0 for key management and secret provisioning. Pilot emerging technologies like Enarx for WebAssembly workloads, Veraison for ARM CCA attestation when hardware becomes available, and MarbleRun for service mesh integration monitoring maturity and community adoption before production deployment.

Training and capability building requires developing internal expertise through certifications like Certified Container Security Expert and Certified Kubernetes Security Specialist, participation in Confidential Computing Consortium working groups and CNCF community meetings, engagement with vendor technical teams at Intel, AMD, ARM, and cloud providers, and establishment of centers of excellence for confidential computing within security and cloud teams. Organizations should budget for specialized consulting during initial deployments and ongoing training as the technology rapidly evolves.

## Conclusion: Confidential computing reaches production maturity with comprehensive ecosystem support

Confidential computing has transitioned from experimental technology to production-ready platform with **formal standardization (IETF RFC 9334, RFC 9397), explicit regulatory recognition as "state of the art" (GDPR, NIS2, DORA), and comprehensive tool ecosystems** supporting attestation, key management, and security testing. The convergence of hardware platforms (AMD SEV-SNP, Intel TDX, ARM CCA), software frameworks (Kata Containers 3.19.1, Gramine v1.4+, Confidential Containers), cloud services (Azure, GCP, AWS Nitro), and regulatory recognition creates an inflection point for widespread adoption.

The most significant insight is that confidential computing solves the fundamental "data-in-use" protection gap that traditional security cannot address, enabling regulatory compliance scenarios previously impossible in cloud environments—particularly **GDPR international transfers post-Schrems II, PCI DSS v4.0 volatile memory protection, and HIPAA data analytics on ePHI**. Organizations waiting for "maturity" risk competitive disadvantage as early adopters establish differentiating capabilities in secure multi-party computation, confidential AI/ML, and privacy-preserving analytics.

Testing protocols have matured with vendor tools (snpguest, Intel Trust Authority, Veraison), formal verification methodologies (ProVerif, CBMC, HOL4), and comprehensive frameworks (IETF RATS architecture, NIST IR 8320 series) providing clear implementation guidance. The gap analysis reveals opportunities for standardized testing frameworks, additional regulatory mappings, and enhanced tooling, but these represent refinement rather than fundamental limitations. Organizations should begin confidential computing pilots immediately, focusing on highest-value use cases like payment processing, healthcare analytics, and cross-border data processing where the technology delivers immediate compliance and security benefits that justify implementation complexity and costs.
