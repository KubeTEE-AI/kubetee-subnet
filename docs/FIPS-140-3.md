# FIPS 140-3 Certification for Confidential Computing and RKE2

## EXECUTIVE SUMMARY

FIPS 140-3 compliance for confidential computing with Rancher RKE2 requires coordinated validation across multiple technology layers, from hardware Trusted Execution Environments (TEEs) to application cryptographic libraries. **The most critical finding: no current TEE implementation has standalone FIPS 140-3 validation**—AMD's Platform Security Processor (Certificate #4723) and Intel's Converged Security Engine (Certificate #4158) provide validated components, but memory encryption and attestation protocols rely on FIPS-approved algorithms without module-level certification.

RKE2 stands out as the **first fully open-source FIPS-validated Kubernetes distribution** (using BoringCrypto Certificate #3678), making it uniquely suited for government and regulated environments requiring both confidential computing and cryptographic compliance. Integration with Kata Containers and Confidential Containers creates an end-to-end secure stack, though organizations must understand validation gaps at each layer.

The transition from FIPS 140-2 to 140-3 accelerates toward the **September 21, 2026 sunset date**, with RHEL 9.2's OpenSSL 3.0.7 FIPS Provider (Certificate #4754) representing the industry's first reusable FIPS 140-3 module. Performance overhead ranges from 15-35% for combined FIPS and confidential computing modes, with I/O-intensive workloads experiencing the highest impact. Organizations requiring FedRAMP High, DoD IL4+, or similar authorizations must implement FIPS at every layer while accepting current architectural limitations in TEE validation.

## Standards evolution drives architectural changes

FIPS 140-3, approved March 22, 2019 and effective for testing since September 22, 2020, represents a fundamental shift from FIPS 140-2's self-contained approach to adopting international standards ISO/IEC 19790:2012 and ISO/IEC 24759:2017. The NIST SP 800-140 series (SP 800-140, 140A through 140F) modifies these ISO standards to create U.S.-specific requirements, introducing enhanced non-invasive security testing (SP 800-140F) and restricting validation to only "Approved" algorithms—eliminating the "Allowed" category that provided 140-2 flexibility.

**The new standard removes hybrid module restrictions**, allowing firmware-hybrid and software-hybrid modules at any security level (140-2 restricted these to Level 1). Software modules at Level 2 no longer require Common Criteria EAL2+ evaluation, instead aligning with CC Operating System Protection Profile requirements. Self-tests evolved from simple Known Answer Tests to conditional execution: Pre-Operational Self-Tests (POST) run at initialization, while Conditional Algorithm Self-Tests (CAST) execute before first algorithm use rather than continuously.

Physical security requirements expanded significantly for Levels 3-4. **Level 3 modules must implement Environmental Failure Testing (EFT) or Environmental Failure Protection (EFP)** for temperature and voltage variations, while Level 4 requires EFP plus fault injection protection. The introduction of a fifth "control output interface" reflects modern architectures where modules generate control signals beyond traditional data flows. These changes particularly impact confidential computing where hardware, firmware, and software boundaries blur across TEE architectures.

FIPS-approved algorithms relevant to confidential computing include AES-XTS (NIST SP 800-38E) for storage encryption, AES-GCM (SP 800-38D) for authenticated encryption, SHA-384 for attestation measurements, and ECDSA P-256/P-384 for digital signatures. The NIST Cryptographic Module Validation Program (CMVP) now uses the Automated Cryptographic Validation Test System (ACVTS) instead of legacy CAVS, with demo access at demo.acvts.nist.gov. Validation timelines extend 12-24 months with costs ranging $50,000-$200,000+ depending on module complexity and security level.

## AMD's PSP provides partial FIPS coverage

AMD achieved FIPS 140-3 validation for Platform Security Processor modules across multiple processor generations, but **only for RSA signature verification and SHA-384 hashing—not memory encryption or attestation signing**. Certificate #4723 (AMD Ryzen PRO 7000 "Phoenix") holds historical significance as the first module validated under NIST's SP 800-140Br1 pilot program on July 11, 2024. Certificate #4626 (Ryzen PRO 6000) and #4401 (Ryzen PRO 5000) cover previous generations, all achieving Level 1 firmware-hybrid validation.

These PSP modules perform critical secure boot functions: verifying the key database and stage 2 firmware integrity using RSA-4096 PKCS PSS signatures with SHA2-384. The cryptographic boundary includes only boot-time verification operations. **AES-128-ECB used for firmware de-obfuscation appears as non-approved** in security policies because it serves non-cryptographic purposes outside CAVP testing scope. No persistent storage of Sensitive Security Parameters occurs—keys exist only in volatile memory, with automatic zeroization via power-off.

SEV-SNP memory encryption uses AES-128 or AES-256 in XEX/XTS modes depending on policy configuration (bit 22: MEM_AES_256_XTS enables enhanced encryption). Each VM receives a unique VM Encryption Key (VEK) generated by hardware RNG in the AMD Secure Processor, stored in dedicated hardware registers inaccessible to software. **This memory encryption implementation is not separately FIPS-validated**, though AMD states compliance with FIPS 140-3 standards. The AES encryption engines reside in memory controllers for transparent encryption/decryption during DRAM operations, with the C-bit (enCrypted bit) in page tables marking encrypted pages.

Attestation relies on Versioned Chip Endorsement Keys (VCEK) derived from chip-unique secrets using NIST SP 800-108 key derivation functions. **VCEK generation combines the Chip Endorsement Key with TCB version** (SNP firmware, PSP bootloader, PSP OS, CPU microcode, and SEV API versions hashed cryptographically) to ensure older firmware cannot generate newer version attestation reports. ECDSA P-384 signatures with SHA-384 sign 1184-byte attestation reports containing launch measurements, chip ID, and TCB metadata. The certificate chain flows: AMD Root Key (ARK, RSA-4096 self-signed) → AMD SEV Key (ASK, RSA-4096) → VCEK (ECDSA P-384).

Versioned Loaded Endorsement Keys (VLEK) provide cloud service provider-specific attestation, introduced in SNP firmware 1.54. CSPs obtain VLEK seeds from AMD Key Distribution Service, encrypted with AES-256-GCM using chip-unique derived keys. The SNP_VLEK_LOAD command loads encrypted VLEKs into AMD-SP. Guest VMs can derive sealing keys via SNP_GET_DERIVED_KEY, mixing measurement, family ID, image ID, policy, security version numbers, and TCB versions to produce 256-bit keys for application use.

**Critical gap: SEV-SNP firmware cryptographic operations lack FIPS validation**. Guest message encryption using AES-256-GCM with VMPCK keys (4 keys per VM for VM privilege levels 0-3), page swap encryption with AES-GCM authentication tags, and attestation report ECDSA signature generation all use FIPS-approved algorithms but operate within non-validated firmware modules. Organizations requiring strict FIPS compliance must document this architectural limitation and evaluate risk acceptance.

## Intel TDX relies on platform components

Intel TDX (Trust Domain Extensions) lacks standalone FIPS 140-3 validation as an architectural feature, instead relying on validated platform components. Intel's Converged Security and Manageability Engine (CSME) holds multiple active FIPS 140-2 certificates (#4648, #4599, #4158, #4150) at Level 1 for hybrid hardware-firmware implementations. **Intel QuickAssist Technology Provider achieved the first Intel FIPS 140-3 validation** (Certificate #5032, Level 1, software-hybrid) with sunset date June 25, 2026.

Multi-Key Total Memory Encryption (MKTME) provides per-Trust Domain encryption using AES-XTS-128 or AES-XTS-256, selected at platform configuration. The encryption engine resides in the direct data path to external memory buses, operating at cache line granularity (64 bytes). **MKTME itself has no FIPS validation**, though it uses FIPS-approved AES-XTS mode specified in NIST SP 800-38E. Trust Domain encryption keys derive using the PCONFIG instruction, unique per TD and unknown to software including the Virtual Machine Manager.

Attestation leverages Intel SGX Data Center Attestation Primitives (DCAP) infrastructure shared between SGX and TDX. **ECDSA-P256 signatures** (FIPS 186-4 compliant) authenticate attestation quotes. The Provisioning Certificate Enclave (PCE) generates platform-unique 256-bit ECC Provisioning Certification Keys (PCK) on NIST P-256 curve, certified to device, SGX TCB SVNs, and PCE's ISVSVN. The Quoting Enclave (QE) generates Attestation Keys using EGETKEY-derived seeds, producing repeatable keys consistent across platform TCB levels.

TDX measurements use **SHA-384 for all 48-byte measurement registers**: MRTD (TD measurement), MRSIGNERSEAM (TDX Module signer), MRSEAM (TDX Module measurement), and four Runtime Measurement Registers (RTMR[0-3]) extended during TD lifecycle. TD Reports include hardware MAC protection using keys accessible only to SGX enclaves via SEAMREPORT instruction, with EVERIFYREPORT2 validating MACs. Quote Version 4 and 5 formats embed PCK certificates (Certification Data Type 5) for simplified verification.

Intel provides extensive entropy source validations under NIST SP 800-90B: Certificate #E57 (Intel DRNG Entropy Source), #E136 (TRNG), #E164 (Digital RNG using metastable latch), and #E179 (DRNG 4). These physical noise sources achieve "Open for Reuse" status, allowing other vendors to incorporate them into validated modules. **AES-NI instruction set acceleration is not separately FIPS-validated**—validation applies to complete cryptographic module implementations using AES-NI, not the CPU instructions themselves. Bouncy Castle FJA 2.1.0 (Certificate #4943, FIPS 140-3 interim) demonstrates successful validation while leveraging Intel AES-NI acceleration.

Intel's approach focuses on making products "FIPS certifiable by design" rather than pursuing end-to-end architectural validation. No "TDX FIPS mode" exists—guest operating systems enable FIPS independently, with TDX providing hardware isolation boundaries. Organizations must architect FIPS compliance at the guest OS and application layers while TDX ensures cryptographic isolation between Trust Domains. Intel Trust Authority provides commercial attestation services but has not published FIPS validation status for its service infrastructure.

## RKE2 delivers first open-source FIPS Kubernetes

Rancher Kubernetes Engine 2 (RKE2) achieved distinction as **the first fully open-source FIPS-validated Kubernetes distribution**, compiled entirely with GoBoring FIPS compiler leveraging BoringCrypto modules (Certificates #2964, #3678, #4156, all FIPS 140-2 Level 1). DISA STIG certification and default CIS Kubernetes Benchmark v1.23/v1.7 compliance position RKE2 for DoD and federal agency deployments requiring both security hardening and cryptographic validation.

FIPS mode activates via single configuration parameter in /etc/rancher/rke2/config.yaml with `fips: true`, automatically configuring all components with validated cryptographic operations. **Every Kubernetes binary compiles with FIPS-enabled toolchain**: API server, kubelet, controller-manager, scheduler, kubectl, containerd, runc, and embedded etcd. The only FIPS-compliant CNI plugin is Canal (default)—Calico and Cilium rebuilds lack FIPS validation despite using approved algorithms.

etcd encryption at rest uses AES-CBC or AES-GCM through EncryptionConfiguration API, with automatic FIPS cipher suite selection when RKE2 operates in FIPS mode. TLS configuration restricts to FIPS-approved cipher suites: TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256, TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384, TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256, and TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384 for TLS 1.2, plus TLS_AES_128_GCM_SHA256 and TLS_AES_256_GCM_SHA384 for TLS 1.3. Minimum TLS version enforcement (TLSv1.2) and certificate management through embedded cert-manager ensure cryptographic chain integrity.

**RKE2 replaces Pod Security Policies with Pod Security Admission** (Kubernetes 1.25+), configurable via pod-security-admission-config-file parameter. Exemptions for confidential computing namespaces allow privileged operations required by Kata Containers while maintaining baseline security posture. Container runtime integration uses containerd with CRI plugin, automatically inheriting FIPS mode from host operating system without additional configuration when host correctly enables FIPS at boot.

Integration with FIPS-validated operating systems requires specific versions: RHEL 8.6+ with OpenSSL 1.1.1k-12.el8_6 (Certificate #3842) or RHEL 9.2+ with OpenSSL 3.0.7 FIPS Provider (Certificate #4754); Ubuntu Pro FIPS 20.04/22.04 with validated OpenSSL 1.1.1; or SUSE Linux Enterprise with FIPS pattern installation. **Host FIPS mode must be enabled during OS installation** using fips=1 kernel parameter—post-installation enablement risks non-compliant key generation during initial boot. Verification requires /proc/sys/crypto/fips_enabled returning 1 and fips-mode-setup --check confirming activation.

## Kata Containers enables confidential workload isolation

Kata Containers provides hardware-virtualized container isolation compatible with RKE2 through containerd shimv2 integration (io.containerd.kata.v2). **FIPS support added November 2019** (Issues #787, #2170) implements automatic host FIPS detection and guest propagation via kernel_params = "fips=1" in /etc/kata-containers/configuration.toml. Guest kernels must compile with CONFIG_CRYPTO_FIPS=y to enable FIPS mode, relying on host-provided FIPS-validated cryptographic modules through inherited libraries.

RuntimeClass configuration defines isolation boundaries: kata-qemu-tdx for Intel TDX with QEMU + TDVF firmware, kata-qemu-sev for AMD SEV-SNP, kata-clh-tdx for Intel TDX with Cloud Hypervisor, and kata-clh for non-confidential virtualized isolation. **Overhead allocations require minimum 250m CPU and 160Mi memory** for kata-agent and VM infrastructure, with production workloads typically allocating 2 vCPUs and 2048Mi base resources plus application requirements.

Confidential Containers operator (v0.10.0+) deploys via kubectl apply with CcRuntime custom resource specifying hardware type and configuration. Installation creates RuntimeClass objects, configures kata-deploy DaemonSet for binary distribution, and initializes attestation infrastructure. The CcNodeSelector matches nodes with appropriate TEE hardware (node.kubernetes.io/instance-type: confidential labels or similar), preventing scheduling on incompatible hardware.

**Attestation architecture implements IETF RATS model**: Attestation Agent (AA) generates evidence inside TEE guest, Key Broker Service (KBS) acts as relying party, Attestation Service (AS) verifies evidence against reference values, and Reference Value Provider Service (RVPS) maintains expected measurements. Evidence includes hardware attestation report (SNP_GET_REPORT or TDX TDREPORT), TEE public key in JWK format, and nonce for freshness. **JWT tokens use RS256 (RSA-PSS + SHA-256)** for signatures, with TLS 1.3 protecting KBS communication using FIPS cipher suites.

Image encryption uses standard OCI image encryption format with Layer Encryption Keys (LEK) wrapped by Key Encryption Keys (KEK) obtained through attestation. The kata-agent image-rs and ocicrypt-rs libraries decrypt layers after successful attestation and key retrieval from KBS. Secret injection occurs runtime via Confidential Data Hub (CDH) intermediary service integrating external KMS providers (HashiCorp Vault, Azure Key Vault, AWS KMS). **LUKS2 disk encryption with AES-256-XTS-PLAIN64** and SHA-256 provides encrypted root filesystems when combined with measured boot attestation.

Performance measurements show 10-20% overhead for Kata isolation alone, 15-25% for confidential computing with SEV-SNP/TDX, yielding 15-35% combined impact depending on workload characteristics. **I/O-intensive workloads experience 25-35% degradation** due to virtio-fs overhead and encrypted memory access patterns. Network throughput decreases 13-16% (iperf3), while CPU-bound operations show 8-14% overhead (sysbench). Optimization strategies include virtio-fs DAX cache enablement, block device allocation for persistent storage, vsock transport for agent communication, and generous resource overallocation.

## Complete stack requires layered validation approach

End-to-end FIPS compliance demands coordinated validation from hardware through application layer, though **no single entity validates the complete confidential computing stack**. Organizations must document and accept gaps while implementing best available validation at each architectural boundary.

**Hardware layer** builds on TPM 2.0 FIPS 140-2 validations (Infineon SLB 9670 Certificate #2959, Microsoft Virtual TPM #4537) providing platform root of trust, cryptographic key generation, and measured boot capabilities. UEFI Secure Boot verifies bootloader signatures before execution, while TPM PCRs store boot measurements for attestation. AES-NI and AVX cryptographic instruction sets provide hardware acceleration without standalone FIPS validation, incorporated into module-level validations for complete implementations.

**Operating system layer** provides validated cryptographic modules essential for all higher layers. RHEL 9.2's OpenSSL 3.0.7 FIPS Provider (Certificate #4754) represents a breakthrough: **first reusable FIPS 140-3 module supporting RHEL 9.4, 9.5, 9.6, and RHEL 10** through modular provider architecture. RHEL 8 maintains OpenSSL 1.1.1k-12 (Certificate #3842), NSS 3.101.0 (#4428), Libgcrypt 1.8.5 (#4384), and Kernel Crypto API 4.18.0-477 (#4397). Ubuntu Pro FIPS provides validated modules for 16.04/18.04/20.04 LTS, with 22.04 submissions for FIPS 140-3 in interim validation as of September 2023.

OS FIPS enablement must occur during installation: grub kernel parameters include fips=1, dracut regenerates initramfs with FIPS modules, and system boots with /proc/sys/crypto/fips_enabled=1. **Post-installation FIPS enablement risks non-compliant cryptographic material generation**—SSH keys, TLS certificates, and encrypted volumes created before FIPS activation may use non-validated algorithms or insufficient entropy. RHEL's fips-mode-setup utility and Ubuntu's ua enable fips command automate configuration but cannot retroactively validate pre-existing secrets.

**Container runtime layer** inherits FIPS mode from host without explicit configuration. containerd automatically detects /proc/sys/crypto/fips_enabled and propagates to container environments via /etc/system-fips mount. CRI-O implements explicit FIPS detection with io.kubernetes.cri-o.DisableFIPS annotation allowing per-pod FIPS exemption. **Docker Enterprise Edition achieved FIPS 140-2 validation (Certificate #3304)** for Swarm state store encryption, secrets management, and Raft log protection. Container images must include FIPS-enabled libraries—registry.access.redhat.com/ubi8-fips provides Red Hat Universal Base Image with validated OpenSSL, NSS, and Libgcrypt.

**Kubernetes layer** cryptographic operations span API server TLS termination, etcd storage encryption, kubelet client authentication, and service mesh communication. EncryptionConfiguration API specifies providers (aescbc, aesgcm, kms) with FIPS-approved algorithms when system operates in FIPS mode. **External KMS integration** enables FIPS 140-2 Level 2+ HSM protection: AWS KMS (Level 2), Azure Key Vault Premium (Level 2), GCP Cloud HSM (Level 3), and Thales Luna HSM (Level 3 FIPS 140-3). Customer-managed keys stored in validated HSMs satisfy highest DoD and federal requirements (IL5 mandates Level 3).

**Application layer** requires FIPS-enabled cryptographic libraries and careful build configuration. OpenSSL 3.x FIPS Provider requires fipsinstall execution generating fipsmodule.cnf, with openssl.cnf configured to activate FIPS and base providers. EVP_default_properties_is_fips_enabled() programmatic check confirms FIPS mode before cryptographic operations. BoringSSL FIPS module (Certificates #3678, #4156) powers Chrome, Android, Google Cloud services, and RKE2 Kubernetes components through BoringCrypto integration.

Go applications present unique challenges: **standard Go crypto library is not FIPS-validated**. Three validated approaches exist: dev.boringcrypto branch using BoringSSL (GOEXPERIMENT=boringcrypto go build), golang-fips/go using OpenSSL via dlopen (GOLANG_FIPS=1), and microsoft/go using Windows CNG on Windows or OpenSSL on Linux. Go 1.24 proposal (#70123) introduces runtime FIPS control via GODEBUG=fips140=on environment variable and crypto/fips140 package for programmatic detection.

## Testing protocols ensure continuous compliance

CAVP (Cryptographic Algorithm Validation Program) provides prerequisite algorithm validation before module-level CMVP certification. **ACVTS (Automated Cryptographic Validation Test System) replaced legacy CAVS** in June 2020, offering demo access at demo.acvts.nist.gov with production access restricted to NVLAP-accredited CST laboratories. Black-box testing generates implementation-specific test vectors matching declared capabilities, validating algorithm correctness without requiring source code disclosure.

CMVP testing covers 11 Derived Test Requirements derived from ISO/IEC 24759:2017 and modified by SP 800-140: (1) Cryptographic Module Specification, (2) Cryptographic Module Interfaces, (3) Roles/Services/Authentication, (4) Software/Firmware Security, (5) Operational Environment, (6) Physical Security, (7) Non-Invasive Security, (8) SSP Management, (9) Self-Tests, (10) Life-Cycle Assurance, (11) Mitigation of Other Attacks. **Testing timeline spans 12-24 months** with lab fees $50,000-$200,000+ depending on module complexity and security level target.

Non-invasive security testing (SP 800-140F) addresses side-channel attacks: timing analysis, differential power analysis, electromagnetic emanation, and fault injection. Level 2+ modules must implement mitigation controls and demonstrate effectiveness through laboratory testing. Self-tests expand beyond Known Answer Tests to include Pre-Operational Self-Tests (POST) at initialization, Conditional Algorithm Self-Tests (CAST) before first algorithm use, pairwise consistency tests (PCT) for asymmetric key generation, and continuous random number generator health tests.

Continuous compliance monitoring requires configuration management preventing algorithm changes triggering revalidation, audit logging of cryptographic operations for compliance review, and periodic recertification addressing discovered vulnerabilities. **OpenSSL FIPS provider 3.0.x maintains validation across minor version updates** through careful change control—adding features outside FIPS boundary while preserving validated module integrity. Organizations must track CMVP Active List updates, monitor for algorithm deprecation (Triple-DES disallowed after 2023, SHA-1 signatures prohibited), and plan migration timelines.

Kata Containers testing specific to FIPS compliance lacks comprehensive automation. Manual verification procedures include: (1) host /proc/sys/crypto/fips_enabled confirmation, (2) kubectl exec into kata pod checking guest /proc/sys/crypto/fips_enabled, (3) attestation log analysis confirming hardware report validation, (4) image decryption success verification, and (5) KBS communication TLS cipher suite inspection. **No FIPS CI/CD test suite exists** in upstream Kata repositories—organizations must develop internal testing covering cryptographic operations, attestation flows, and performance regressions.

## Federal requirements span multiple frameworks

FedRAMP (Federal Risk and Authorization Management Program) mandates FIPS 140-validated or NSA-approved cryptography across all impact levels (Low, Moderate, High). **SC-13 (Cryptographic Protection) control requires FIPS-validated modules for all cryptographic functions**: data-at-rest encryption, data-in-transit protection, authentication, digital signatures, hashing, and key generation. FedRAMP High baseline specifies 421 controls including mandatory FIPS 140-2/140-3 encryption, multi-factor authentication using validated modules (NIST SP 800-63B), and audit log protection with cryptographic integrity.

DoD Cloud Computing Security Requirements Guide (CC SRG) defines impact levels IL2 through IL6 with escalating requirements. **IL4 (Controlled Unclassified Information)** requires FIPS 140-2/140-3 validated encryption for data-at-rest and in-transit, NIPRNet connectivity via Cloud Access Point (CAP), US persons for support, and DoD-approved locations. **IL5 (CUI requiring higher protection, National Security Systems)** mandates FIPS 140-2 Level 3 HSMs, IL5-adjudicated US persons only, dedicated DoD private or Federal community clouds, customer-managed keys in FIPS-validated HSMs, and US/US-outlying area locations in DoD facilities. IL6 (classified up to SECRET) adds SIPRNet connectivity, cleared/classified facilities, and highest cryptographic validation levels.

NIST SP 800-53 Rev 5 control families relevant to confidential computing include SC-12 (Cryptographic Key Establishment and Management) requiring FIPS-validated or NSA-approved key management technology, SC-28 (Protection of Information at Rest) specifying FIPS-validated encryption mechanisms, SC-8 (Transmission Confidentiality and Integrity) mandating FIPS cryptographic protection, and IA-5 (Authenticator Management) governing cryptographic authenticators. **SC-12 enhancements address specific scenarios**: SC-12(1) availability through key backup and recovery, SC-12(2) symmetric key production with FIPS-approved RNGs, and SC-12(3) asymmetric key generation and distribution following FIPS 186-5.

Common Criteria (ISO/IEC 15408) provides complementary evaluation focusing on overall IT product security rather than cryptographic module validation specifically. **NIAP Policy Letter #5 (December 6, 2019) clarifies relationship**: products may pursue both FIPS and CC certifications addressing different security aspects. Trustonic Kinibi TEE achieved EAL5+ (highest for Secure OS), while Qualcomm TEE Software Cryptographic Library obtained FIPS 140-2 Certificate #4322 (Level 1, Software-Hybrid). Organizations pursuing FedRAMP High or DoD IL5+ typically require both FIPS cryptographic validation and CC evaluation demonstrating security functionality.

International standards harmonization includes ISO/IEC 19790:2012 (security requirements for cryptographic modules) adopted as FIPS 140-3 normative reference and ISO/IEC 24759:2017 (test requirements) providing testing methodology modified by SP 800-140 series. Cloud service providers navigating international deployments benefit from FIPS alignment with ISO standards—AWS, Azure, and GCP maintain FIPS-validated cryptographic modules plus Common Criteria evaluations enabling global compliance frameworks. **Federal agencies with international operations must address FIPS-only requirements conflicting with other nations' cryptographic standards**, requiring careful architecture supporting multiple validation regimes.

## Implementation requires careful planning and validation

Step-by-step FIPS deployment for RKE2 with confidential containers begins with hardware selection: AMD EPYC 7xx3+ (Milan, Genoa) for SEV-SNP or Intel Xeon Scalable 4th/5th Gen (Sapphire Rapids, Emerald Rapids) for TDX. **Verify CPU SKU supports confidential computing features**—not all processors in generation include SEV-SNP or TDX. BIOS configuration must enable secure boot, configure SEV-SNP/TDX in platform setup, verify TPM 2.0 activation, and confirm nested virtualization or bare metal deployment.

Operating system installation must enable FIPS during initial setup appending fips=1 to kernel parameters—post-installation enablement risks compliance gaps. RHEL 9.2+ provides optimal experience with OpenSSL 3.0.7 FIPS Provider reusability, while Ubuntu Pro FIPS 22.04 offers FIPS 140-3 interim modules submitted September 2023. **Verification sequence confirms proper configuration**: cat /proc/sys/crypto/fips_enabled returns 1, fips-mode-setup --check reports FIPS mode enabled, and openssl list -providers shows FIPS provider active.

RKE2 installation with FIPS requires /etc/rancher/rke2/config.yaml configuration specifying fips: true, profile: cis-1.6, selinux: true, and secrets-encryption: true. **First node becomes RKE2 server** (systemctl enable rke2-server && systemctl start rke2-server), with worker nodes joining using server URL and token from /var/lib/rancher/rke2/server/node-token. Verification checks kubectl get nodes confirming Ready status and kubectl get pods --all-namespaces showing all system components Running.

Kata Containers deployment via kata-deploy DaemonSet automates binary distribution and containerd configuration: kubectl apply -f kata-rbac/base/kata-rbac.yaml establishes permissions, kubectl apply -k kata-deploy/overlays/rke2 adapts configuration for RKE2's containerd location (/var/lib/rancher/rke2/agent/etc/containerd/), and kubectl apply -f kata-deploy/base/kata-deploy.yaml initiates installation. **Configuration.toml modification specifies FIPS kernel parameters**: kernel_params = "fips=1 agent.log=debug" propagates FIPS mode to guest VMs.

Confidential Containers operator deployment (v0.10.0+) creates CcRuntime custom resource specifying hardware type (AMD SEV-SNP or Intel TDX) and node selector labels. Trustee attestation infrastructure requires separate namespace (coco-tenant or operators), TLS certificate generation for KBS HTTPS endpoint, authentication key pair creation for JWT token signing, and ConfigMap preparation with attestation service configuration. **Reference value population in RVPS** must include expected measurements: kernel SHA-384 hash, initrd measurement, firmware version for SEV-SNP, or MRTD/RTMR values for TDX.

Network configuration selection impacts FIPS compliance: Canal (RKE2 default) maintains FIPS validation, while Calico and Cilium rebuilds lack validated modules despite using approved algorithms. Pod Security Admission configuration exempts confidential computing namespaces (kube-system, confidential-containers-system, operators) from restricted enforcement while maintaining baseline security. **NetworkPolicy resources isolate confidential workloads** allowing only DNS (UDP 53) to kube-system and KBS communication (TCP 8080) to operators namespace, blocking all other ingress/egress.

Common pitfalls include firewalld conflicts with Canal/Calico requiring firewall disablement, NetworkManager interference with Kata interfaces needing manual exclusion configuration, SELinux contexts preventing kata-runtime execution requiring permissive mode or custom policies, and nested virtualization detection failures when CPU flags (svm for AMD, vmx for Intel) lack propagation to nested guests. **Image sharing limitations prevent host-level caching for encrypted images**—each pod independently retrieves and decrypts images, increasing network bandwidth and startup latency.

Performance optimization strategies balance security requirements with operational needs. **Resource overallocation compensates for FIPS and CC overhead**: minimum 2 vCPUs and 2048Mi RAM baseline per Kata pod plus application requirements, virtio-fs with DAX cache enabled (virtio_fs_cache = "always", virtio_fs_cache_size = 8192), block device allocation for I/O-intensive workloads rather than virtio-fs, and vsock transport for kata-agent communication reducing network stack overhead. Monitoring reveals actual overhead patterns enabling right-sizing: CPU-bound workloads tolerate 8-14% overhead, memory operations experience 12-17% reduction, I/O-intensive applications suffer 25-35% degradation, and network throughput decreases 13-16%.

Migration from non-FIPS to FIPS environments demands comprehensive planning spanning 6-8 weeks for gradual production migration. **Phase 1 assessment** inventories current workloads identifying FIPS-incompatible components (custom cryptographic implementations, deprecated algorithms, hardcoded keys), while Phase 2 preparation enables FIPS on test clusters and rebuilds container images with validated libraries. **Phase 3 migration** follows gradual rollout: non-production namespaces first, progressive production workload migration with careful monitoring, and immediate rollback capability for detected issues. Phase 4 validation runs comprehensive FIPS verification suites, performance benchmarking against baseline, and attestation flow testing under production loads.

Vendor roadmaps guide long-term planning: AMD targets complete KVM SEV-SNP upstreaming Q1 2025 with vTPM support Q2 2025 and FIPS 140-3 PSP certification 2026; Intel progresses TDX availability on Emerald Rapids Q1-Q2 2025 with full KVM upstreaming Q3 2025 and TDX 2.0 enhanced features planned; Rancher maintains RKE2 FIPS support v1.28+ with enhanced testing framework v1.30+; Kata Containers community prioritizes enhanced FIPS testing framework 2025 and native FIPS module integration for Rust-based attestation agent long-term. **Monitor NIST CMVP for certificate updates** tracking OpenSSL 3.x revalidations, BoringSSL transitions to FIPS 140-3, and new TEE component validations.

## Known gaps require risk acceptance

Organizations must document and accept architectural limitations where FIPS validation gaps exist. **No current TEE implementation holds standalone FIPS 140-3 validation**—AMD PSP validates only boot-time RSA signature verification and SHA-384 hashing (Certificates #4401, #4626, #4723), while Intel CSME validates platform cryptographic operations (Certificates #4158, #4150) but not TDX-specific memory encryption or attestation protocols. Memory encryption relies on FIPS-approved algorithms (AES-XTS, AES-GCM) without cryptographic module validation of complete implementations.

Attestation cryptography uses validated algorithms—ECDSA P-256/P-384 for signatures, SHA-256/384 for measurements, RSA-2048/4096 for certificate chains—but **attestation firmware lacks FIPS module certification**. AMD SEV-SNP firmware signing attestation reports with VCEK/VLEK and Intel TDX Module generating TD Reports with hardware MAC protection operate within non-validated firmware boundaries. Guest message encryption in SEV-SNP using AES-256-GCM with VMPCK keys and page swap encryption with authenticated tags similarly lack validation despite algorithm compliance.

Kata Containers components inherit validation from underlying OS cryptographic libraries but **kata-runtime and kata-agent binaries themselves are not FIPS-validated**. Go crypto in kata-runtime requires FIPS-enabled toolchain (GoBoring, golang-fips/go, or microsoft/go) which upstream Kata builds may not use. Rust cryptographic crates in kata-agent (image-rs, ocicrypt-rs) lack comprehensive FIPS validation—some crates use FIPS-validated OpenSSL bindings while others implement algorithms in Rust without validation.

**Confidential Containers attestation agent cryptography presents validation gaps**: Rust crypto libraries performing JWT token generation, ephemeral key pair creation for secret wrapping, and TLS communication to KBS may not use FIPS-validated implementations despite selecting FIPS-approved algorithms. Organizations requiring strict FIPS compliance must either accept risk, contribute upstream FIPS-enabled builds, or develop internal validated attestation solutions.

Performance trade-offs compound with layered security: FIPS mode alone adds 8-15% overhead, confidential computing (SEV-SNP/TDX) adds 5-15%, and combined operation yields 15-35% total impact with highest degradation in I/O-intensive workloads. **Capacity planning must account for security overhead**—1000-node non-FIPS cluster may require 1150-1350 nodes achieving equivalent throughput with full FIPS+CC enablement. Storage I/O suffers most (25-35% reduction) from encrypted memory access patterns and virtio-fs overhead, while CPU and network show moderate impact (8-16%).

Testing coverage gaps require internal investment: **no comprehensive FIPS CI/CD suite exists** in upstream Kata Containers or Confidential Containers projects. Organizations must develop testing covering cryptographic algorithm validation (confirming FIPS-approved algorithms active), attestation protocol verification (testing full evidence generation through secret release flow), performance regression detection (establishing baselines and monitoring degradation), and compliance audit automation (continuous validation of FIPS mode across stack layers).

Cloud provider FIPS implementations vary in comprehensiveness: AWS achieves FIPS 140-3 validation for AWS-LC cryptographic library (Certificate #4631) with CloudHSM providing Level 3 hardware protection; Azure maintains Windows cryptographic modules with FIPS 140-2 validation transitioning to 140-3, offering Managed HSM Level 3 for customer-managed keys; GCP uses BoringCrypto (Certificates #3678, #3900) noting BoringSSL wrapper lacks validation though BoringCrypto module holds certificates. **DoD provisional authorizations (IL2/IL4/IL5) do not guarantee full FIPS validation at all architectural layers**—authorizing officials accept risk based on overall security posture rather than complete cryptographic validation.

FIPS 140-2 sunset September 21, 2026 creates migration urgency: all 140-2 certificates transition to Historical list requiring revalidation under FIPS 140-3. **Organizations must plan dual transitions**: (1) enabling FIPS if not currently implemented, and (2) migrating 140-2 modules to 140-3 validated replacements. RHEL 9.2+ with OpenSSL 3.0.7 FIPS Provider (Certificate #4754) provides forward path through reusability across RHEL 9.4-10, while Ubuntu Pro FIPS 22.04 interim submissions await final certification. Delayed migration risks operational disruption when authorizing officials require active 140-3 certificates for continued production authorization.

## Conclusion: Navigate validation gaps with informed decisions

FIPS 140-3 compliance for confidential computing with RKE2 requires accepting architectural limitations while maximizing validation coverage at achievable boundaries. No complete end-to-end FIPS validation exists—TEE hardware provides FIPS-approved algorithms without module certification, RKE2 delivers validated Kubernetes with BoringCrypto integration, operating systems supply validated cryptographic libraries, and applications must implement FIPS-enabled toolchains. Organizations requiring FedRAMP High, DoD IL4+, or equivalent authorizations can achieve approval through documented risk acceptance of validation gaps combined with defense-in-depth security architecture.

The transition from FIPS 140-2 to 140-3 before September 21, 2026 sunset creates immediate planning requirements. RHEL 9.2's reusable OpenSSL FIPS Provider demonstrates maturation of modular validation approaches, while vendor roadmaps (AMD PSP certification 2026, Intel component validations ongoing) suggest gradual TEE validation improvement. Performance overhead of 15-35% for combined FIPS and confidential computing modes demands capacity planning and workload profiling, with I/O-intensive applications requiring careful optimization or hardware overprovisioning.

RKE2 with Kata Containers and Confidential Containers provides production-ready confidential computing for government and regulated workloads requiring hardware-based isolation combined with cryptographic compliance. Implementation demands careful hardware selection, OS FIPS enablement during installation, coordinated component configuration across stack layers, and comprehensive validation testing. Organizations should monitor vendor FIPS roadmaps, contribute upstream testing improvements, and architect migration strategies enabling gradual production adoption with rollback capabilities. Success requires balancing security requirements against operational constraints while documenting accepted risks where validation gaps exist.
